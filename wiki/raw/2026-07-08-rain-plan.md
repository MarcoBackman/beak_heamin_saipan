# 우천 플랜 병기 — 날씨 적응형 일정

## 배경 및 목표

day별 날씨(예보/평년)에 따라 우천 시 대체 액티비티를 일정에 병기한다. 항공편·로타
경비행기·마나가하 페리 등 예약이 고정돼 있고 실예보는 출발 약 16일 전부터만 제공되므로,
일정 자체를 자동 변경하지 않고 **큐레이션된 우천 플랜을 나란히 보여주고 현장에서 두 사람이
결정**하는 방식을 택한다 (사용자 확정).

**목표**
1. D2~D7 각 day 일정 아래 '☔ 우천 시 플랜' 접이식 블록을 병기한다 (D1 심야 도착·D8 새벽 출국 제외).
2. 해당 day의 해석된 날씨(예보 또는 9월 평년)가 비/뇌우면 블록을 자동 펼침 + 시각 강조하고,
   day 헤더에 "☔ 우천 플랜 확인" 미니 칩을 띄운다.
3. JS 실패·인쇄·오프라인에서도 우천 플랜 콘텐츠에 접근 가능해야 한다.

**성공 기준**: 현재(평년 폴백) 기준 D3·D7(rain)·D5(thunder)가 자동 펼침+강조 상태로 보이고,
sun/cloud day는 접힌 기본 상태. JS 차단 시에도 수동 개폐 동작. 인쇄물에 우천 플랜 포함.
기존 기능 회귀 0.

## 선택한 접근과 이유

**정적 HTML 접이식 병기 블록 + weather.js 최소 연동** (2단계 객관식으로 확정)

- 콘텐츠는 index.html의 각 day에 `<details class="rain-plan">`로 정적 추가 — 일정 본문이
  index.html에 있는 기존 콘텐츠 소유 구조와 일치, JS 무관하게 동작.
- weather.js `renderDay()`에 소규모 추가 — 테마가 `rain|thunder`면 `.rain-plan` 자동 펼침 +
  ☔ 칩. 별도 신규 JS 파일 없음.
- 강조 스타일은 `article.day[data-wx=rain] .rain-plan` 등 CSS 셀렉터만으로 처리(배너 역할).

## 아키텍처

- **index.html** — D2~D7 `.sched` 아래(참고 콘텐츠 블록 위) `details.rain-plan` 추가:
  `<summary>☔ 우천 시 플랜</summary>` + `<ul>` 불릿 2~4개.
- **js/weather.js** — renderDay에서 theme이 rain/thunder일 때:
  ① `art.querySelector('.rain-plan')`이 있으면 `open = true`
  ② `.day-head`에 `.wx-rain-chip`("☔ 우천 플랜 확인") append. 부재 시 no-op.
- **css/weather.css** — `.rain-plan` 기본 스타일(기존 details.practical·fold와 시각 언어 통일),
  `[data-wx=rain] .rain-plan, [data-wx=thunder] .rain-plan` 강조(테두리·배경 틴트), `.wx-rain-chip`.
- **인쇄** — `window.addEventListener('beforeprint', ...)`로 `.rain-plan` 강제 펼침(weather.js에
  2줄 — printBtn과 브라우저 Ctrl+P 모두 커버). 우천 플랜은 오프라인·인쇄에서 가치가 크므로
  인쇄물에 포함한다.
- **캐시** — 수정 파일 index.html 내 weather.js/weather.css `?v=` 증가.

## 데이터 흐름

```
weather.js: 예보/평년 해석 → data-wx (기존)
          → theme rain|thunder → .rain-plan.open = true + ☔ 칩 append
CSS: data-wx 기반 .rain-plan 강조 (JS와 무관하게 동작)
사용자: 어느 날이든 details 수동 개폐 가능
```

## 인터페이스 (모듈 경계)

- 신규 JS API 없음. 계약은 두 가지뿐:
  - `data-wx` 속성 (기존 — weather.js 생산, weather.css 소비)
  - `.rain-plan` 클래스 (index.html 생산, weather.js·weather.css 소비)
- weather.js는 `.rain-plan` 부재 시 아무것도 하지 않음 (D1·D8 안전).

## 에러 처리

- weather.js 로드 실패 → 접이식 블록은 정적 HTML이라 그대로 사용 가능 (강조·자동 펼침만 없음).
- 예보 없음(현재~8월) → 평년 테마 기준 동일 동작.
- data.js 실패 → weather.js 가드로 종료(기존 패턴), 블록은 정적으로 잔존.

## 우천 플랜 콘텐츠 (확정 초안 — 사용자 승인됨)

공통 원칙(첫 번째 우천 블록 D2 상단에 1줄 반영): 비 자체는 수영·다이빙에 지장 적음(어차피
젖음) — 문제는 **번개(입수 금지), 파도(보트 결항), 진창(트레킹)**. 9월 사이판 스콜은 대개
30분~1시간이라 "대피 후 재개"가 1순위.

| Day | 우천 시 플랜 |
|---|---|
| D2 그로토·라우라우 | 스콜이면 순서 스왑(라우라우↔그로토) 후 재개. 번개면 입수 중단 → 아메리칸 메모리얼 파크 비지터센터(실내 전시), 조텐 마트 간식 장보기, 호텔 스파. 그로토 적기 게양 시 무조건 라우라우로 |
| D3 로타홀 보트 | 파도로 보트 취소 시 → 토가 동굴(실내 동굴, 우천 무관), 송송 빌리지 산책·카페, 피나탕 파크(스웰 없으면 비 와도 가능). 빛내림은 D4 오전 재시도 여지(업체 협의) |
| D4 테테토·스위밍홀 | 우천 시 스위밍홀 비포장 진입로 진창 → 스킵하고 테테토만(번개만 아니면 수영 가능). 남는 시간 토가 동굴·송송 카페. 오후 비행이라 무리하지 않기 |
| D5 마나가하 | 페리는 웬만한 비엔 운행, 결항 시 → 가라판 데이로 전환: 아이러브사이판 쇼핑(D7 앞당기기), 아메리칸 메모리얼 파크, 카페 투어. 섬에서 스콜은 파빌리온 대피 후 재개 |
| D6 포비든 트레킹 | 우천 시 투어 취소가 보통(진창·미끄럼) → D7 윙 비치와 통째로 스왑 시도, 또는 마이크로 비치(라군은 비에도 잔잔) + 쇼핑 선행 |
| D7 윙비치·가라판 | 윙 비치 옵션 스킵(파도 있는 날 원래 금지). 쇼핑·디너는 우천 무관. 선셋 크루즈 취소 시 가라판 디너 연장 |

## 테스트 전략

브라우저 실동작 검증 (기존 컨벤션 — 테스트 프레임워크 없는 정적 페이지):

1. **390px/1280px** — D2~D7 블록 렌더, 평년 폴백 기준 D3·D5·D7 자동 펼침+강조+☔ 칩,
   sun/cloud day 접힘·비강조, 콘솔 에러 0
2. **JS 차단** — weather.js 미로드 상태에서 블록 수동 개폐 동작
3. **인쇄 미리보기** — 우천 플랜 내용 포함(강제 펼침), 날씨 연출 레이어는 기존대로 숨김
4. **회귀** — 배지·배경 연출·지도 오버레이·접힘(fold) 무영향, ResizeObserver와 상호작용
   (블록 펼침 → 카드 높이 변화 → 캔버스 추적) 정상
5. **배포** — Pages 실URL 확인

## 리스크

- 🟡 콘텐츠 사실성 — 토가 동굴 상시 개방 여부, 아메리칸 메모리얼 파크 비지터센터 운영시간
  등은 구현 단계에서 웹 검색으로 확인 후 확정. 불확실하면 문구를 보수적으로("~가능 여부 현지 확인")
- 🟡 로타 경비행기 악천후 지연 — 일정 변경이 아닌 안내 문구로만 반영(예약 고정)
- 🟢 회귀 — 순수 추가 변경. 단 rain-plan 펼침이 카드 높이를 바꾸므로 날씨 캔버스가 따라와야
  함 — 직전 수정(ResizeObserver)으로 커버됨, 테스트 4에서 확인

## 기각된 접근

- **예보 기반 자동 스왑** — 예보 불확실성·예약 고정과 충돌, 원래 일정이 숨겨져 혼란 → 기각
- **토글 스왑(기본↔우천 교체)** — 두 플랜 동시 비교 불가, 우천판 전체 일정 별도 작성 부담 → 기각
- **별도 '우천 대비' 통합 섹션** — 해당 날짜를 볼 때 왔다갔다 필요 → 기각
- **data.js 데이터 주도 렌더** — 일정 본문이 index.html에 있는 기존 소유 구조와 불일치,
  이동 파트가 늘어남 → 기각 (YAGNI)
